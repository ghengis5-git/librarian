# Release Notes — v0.8.0

**Date:** 2026-04-22
**Version:** V6.0
**Status:** Draft

## Overview

**Minor-version release.** Bundles three independent workstreams completed since v0.7.5: (1) a Codex-driven security fix to the evidence-seal pipeline, (2) Phase 8.2a adoption/hardening work (new legal-discovery template, friendly YAML parse errors, Windows/UNC path hardening), and (3) a four-pass adversarial code review with two HIGH-severity fixes (UnicodeDecodeError handling in the YAML loader, and silent evidence-chain corruption from duplicate rel_path resolution).

**No breaking changes, no schema migrations, no registry edits required** — except where noted under "Upgrade guidance".

**Recommended for all users on v0.7.5 or earlier.** Two of the fixes (evidence-seal time-dependency, duplicate-rel_path silent attestation) affect the tamper-evident IP evidence pack — the load-bearing output of the tool for audit and compliance workflows. Existing evidence packs generated by v0.7.5 or earlier still verify against the signing SSH key, but any evidence regeneration will now produce a stable canonical seal that survives re-verification after arbitrary delay.

Minor version bump (not patch) because `librarian.yaml_errors` is a new public module and the manifest/evidence canonical form changed shape.

## What's in it

### Track 1 — Evidence-seal security fixes (from Codex adversarial review 2026-04-15)

Two HIGH-severity bugs in the tamper-evident evidence pack. Fixed in commit `5bc0b16`.

#### HIGH

- **Evidence seal time-dependency (`evidence.py`).** `generate_evidence()` sealed `manifest.to_json()`, which includes `generated_at`. `verify_evidence()` then regenerated a fresh manifest and re-hashed it, producing a different seal every time — effectively at least one second after generation, depending on clock granularity. Evidence packs were single-use verification tokens. **Fix:** `Manifest.to_canonical_dict()` / `to_canonical_json()` exclude `generated_at`. Both `generate_evidence()` and `verify_evidence()` seal over the canonical form. Regression test `test_seal_stable_after_delay` sleeps 1.1s between generate and verify — must pass.

- **Manifest basename collision silently attests wrong file (`manifest.py`).** `_hash_file()` stored `path.name` (basename) in `FileHash.filename`. `_resolve_file_path()` returned the first `rglob()` match. If two tracked directories contained the same basename, the wrong file was hashed and the seal attested the wrong content with no error raised. **Fix:** `_hash_file()` now accepts `registered_path` and stores the relative path from `repo_root` (e.g. `docs/alpha.md`) instead of the bare name. `generate()` detects basename collisions across tracked dirs and raises `ManifestError` before any hashing occurs. Missing files fall back to the registry `path` field or registered filename.

Related changes:
- `ManifestError` exported from `librarian.__init__`.
- `EvidencePack.generator_version` bumped 0.7.0 → 0.8.0.
- `seal.note` updated to reference canonical form.

### Track 2 — Phase 8.2a: adoption helpers + platform hardening

Three independent items shipped in commit `283c242`.

#### New template — `legal-discovery`

New `librarian/templates/legal/legal-discovery.md` matching the format of the existing six legal templates. Eleven sections, cross-refs to `legal-review`, `contract-summary`, `nda-tracker`, compliance conditionals for HIPAA (PHI handling in discovery), SEC/FINRA (Rule 17a-4, FINRA 4511), and DoD 5200 (classification/CUI). Ends with Attorney Work Product privilege notice.

Integration: `EXPECTED_LEGAL_IDS` in `tests/test_templates.py` updated; `test_legal_template_count` bumped 6 → 7; `PRESET_EXPECTATIONS["legal"]["recommended"]` appended so the legal preset auto-suggests it.

#### New public module — `librarian.yaml_errors`

Friendly YAML parse errors. New `YamlParseError(Exception)` subclass with structured `.path / .line / .column / .problem` attributes. The original `yaml.YAMLError` is chained via `raise ... from e` so tracebacks still show parser state.

The module formats parse errors with the exact source line and a caret-pointer under the offending column:

```
docs/REGISTRY.yaml:12:3: YAML parse error
  expected <block end>, but found '<scalar>'
  12 | - filename: docs/foo.md
     |   ^
```

Public API:
- `YamlParseError` — exception subclass with structured attrs.
- `load_yaml(path)` — wraps `yaml.safe_load` for file loads.
- `load_yaml_string(source, source_label="<string>")` — for in-memory YAML (frontmatter, embedded config blocks).

All exported from `librarian.__init__`. Integration points:
- `librarian/registry.py::Registry.load()` now routes through `load_yaml`.
- `librarian/config.py::load_defaults_file` now routes through `load_yaml`.

Hardening choices baked in from in-session adversarial review:
- **Tab-safe caret.** `_caret_prefix` mirrors tab characters in the source line so the caret aligns correctly in any terminal, regardless of tab-width rendering.
- **Bounded source-line re-read.** `itertools.islice` with `_SOURCE_LINE_LOOKAHEAD = 5` caps the re-read window so a malformed multi-gigabyte YAML file can't force a full-file allocation.
- **CRLF tolerance.** `rstrip("\r\n")` instead of `rstrip("\n")` so Windows-authored files don't leak trailing `\r` into the formatted message.

#### Windows / UNC path hardening (`precommit.py`)

Three real failure modes on Windows that the POSIX-only code didn't handle:
1. Mapped-drive → UNC resolution mismatch (`Path.resolve()` converts `Z:\proj` to `\\server\share\proj`).
2. Case-insensitivity (NTFS compares case-insensitively; `Path.relative_to` is case-sensitive on every platform).
3. Disconnected shares (`Path.resolve()` on offline UNC can raise `OSError`).

Fix: module-level `_IS_WINDOWS = os.name == "nt"` gate. New helpers `_norm_key(p)` (normalizes case + forward slashes on Windows; no-op on POSIX) and `_safe_resolve(p)` (wraps `Path.resolve` with `except OSError` fallback to the original path).

`_find_registry` rewritten to use `_safe_resolve` + `_norm_key` + a `seen` set to defend against pathological idempotent-parent behavior on UNC anchors. `_should_check` containment check routed through the new helpers with a case-insensitive fallback when `relative_to` still disagrees on case.

Zero behavior change on POSIX — all new logic gated on `_IS_WINDOWS`. Regression test `TestShouldCheckWindowsCaseInsensitive.test_case_sensitive_on_posix_gate` explicitly asserts the gate holds.

Not validated on actual Windows (no host available) — logic is verified via `monkeypatch` on `_IS_WINDOWS` and `os.path.normcase`.

### Track 3 — Four-pass adversarial review

Every minor-version bump includes an adversarial code review before ship. This release went through four passes. Passes 1 and 2 (self-review during initial work) caught 6 issues (M1, M2, L1–L4) — all fixed in-session. Pass 3 (Codex subagent) found two additional HIGHs. Pass 4 (Opus subagent static-analysis audit) found two more. All four HIGHs fixed before release.

#### HIGH — Pass 3

- **Caret off-by-one (`yaml_errors._caret_prefix`).** The helper sliced `source_line[:max(column, 0)]` (padding *through* the offending column) instead of `source_line[:max(column - 1, 0)]` (padding *up to but excluding* it). Every reported error pointed one column too far right. Fix: `pad_chars = max(column - 1, 0)` so `column=1` yields an empty prefix (caret flush at column 1). Existing unit tests had actually encoded the bug (docstring said "column 1 → no prefix" while asserting one character of pad) and were corrected alongside the fix. New `test_caret_lands_under_correct_column` asserts the caret character sits at the reported column using a realistic source+caret pair.

- **Missing `UnicodeDecodeError` catch in precommit (`precommit._load_project_config`).** The function opened the registry with `encoding="utf-8"` but only caught `OSError` and `yaml.YAMLError`. A non-UTF-8 registry (UTF-16 BOM, stray high-bit bytes) raises `UnicodeDecodeError` before PyYAML runs, crashing the pre-commit hook instead of honoring its non-blocking contract. Fix: added `UnicodeDecodeError` to the exception tuple. Regression tests in `tests/test_precommit.py::TestLoadProjectConfig`.

#### HIGH — Pass 4

- **`UnicodeDecodeError` also missing in `yaml_errors.load_yaml`.** Same failure mode at a different entry point. A UTF-16-BOM or Latin-1 registry raised `UnicodeDecodeError` from `open(..., encoding="utf-8")` before PyYAML ran, giving the user a raw Python traceback with no path context — the exact failure mode this wrapper was built to prevent. Fix: added `except UnicodeDecodeError` branch that re-raises as `YamlParseError` with byte offset, decode reason, and "re-save as UTF-8 without BOM" guidance. Five regression tests in `tests/test_yaml_errors.py::TestLoadYamlUnicodeDecodeError`.

- **Silent evidence-chain corruption from duplicate `rel_path` resolution (`manifest.py::generate`).** Companion bug to the basename-collision fix from commit `5bc0b16`. If two registry entries share a `filename` and neither has an explicit `path:` field, `_resolve_file_path` walks `tracked_dirs` and returns the *first* match for both. The `file_hashes` list then contains the same file hashed twice, and the second registration's real file (which may be on disk under a different path) is silently dropped from the evidence chain — any tamper to that shadowed file is undetectable. Fix: `seen_rel_paths` dict check in both the on-disk and missing-file branches of the Type-2 loop. Any repeat `rel_path` now raises `ManifestError` with guidance to add distinct `path:` entries. The existing `seen_basenames` check is preserved — it fires when basenames collide with *different* `rel_path` (rename-based collision across tracked dirs); the new check is its inverse (same `rel_path` registered twice). Four regression tests in `tests/test_manifest.py::TestDuplicateResolutionDetection`, including an end-to-end tamper-detection proof.

#### MEDIUM and LOW

All MED and LOW findings from pass 1 (M1, M2) and pass 2 (L1, L2, L3, L4) were remediated in-session during the initial Phase 8.2a work. See the commit log and CLAUDE.md "Session 53 Deliverables" for the full trail.

## Upgrade guidance

- **From v0.7.5:** `pip install --upgrade librarian-2026`. No migration steps, no registry changes.

- **If you previously generated evidence packs with v0.7.5 or earlier:** existing packs still verify against the original manifest JSON. Regenerating evidence under v0.8.0 produces a new `seal` value because the seal now covers the canonical form (no `generated_at`). The SSH-signature chain is unchanged. Plan on one evidence regeneration as part of upgrading.

- **If you have two registry entries with identical `filename` and no explicit `path:` field:** v0.8.0 will now raise `ManifestError` during `librarian manifest`. Resolution: add a distinct `path:` to each entry. This was previously a silent evidence-chain corruption.

- **If you import from the new `librarian.yaml_errors` module:** public API is stable. `YamlParseError`, `load_yaml`, and `load_yaml_string` are exported from `librarian` top-level.

- **Pre-commit framework users:** bump the `rev:` in your `.pre-commit-config.yaml` to `v0.8.0`, or run `pre-commit autoupdate`.
